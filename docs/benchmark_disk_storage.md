# ディスクストレージ ベンチマークレポート

実行日: 2026-02-28
環境: Apple M2 Pro, darwin/arm64, Go

## 概要

disk ストレージ (`--storage disk`) と memory ストレージ (デフォルト) の性能差を、クエリパターン別に計測した。
disk ストレージはデータをページ単位でディスクに格納し、バッファプール (LRU 256 ページ) でホットページのみメモリに保持する。
memory ストレージのセカンダリインデックスはメモリ上の B-tree、disk ストレージのセカンダリインデックスは同一 `.db` ファイル内にページベースの B+Tree として永続化される（DiskSecondaryBTree）。disk ではインデックスの検索自体もバッファプール経由のページフェッチとなるため、ポイントルックアップにはオーバーヘッドが加わるが、起動時の全行スキャンによるインデックス再構築が不要になる。

## ベンチマーク結果

### 1. PK / セカンダリインデックス ポイントルックアップ (10,000行)

| パターン | Memory (ns/op) | Disk (ns/op) | Disk/Memory | Memory (B/op) | Disk (B/op) |
|---------|---------------|-------------|-------------|--------------|------------|
| PK lookup (`WHERE id = X`) | 882 | 3,054 | **3.5x** | 920 | 8,008 |
| セカンダリインデックス (`WHERE val = X`) | 1,087 | ~~3,048~~ 6,923 | ~~**2.8x**~~ **6.4x** | 1,050 | ~~8,138~~ 14,770 |

**考察:** PK ルックアップは disk で 3.5x 遅くなる。主な原因はバッファプールのページフェッチ + 行デコードのオーバーヘッド。セカンダリインデックスのルックアップはセカンダリインデックスのディスク永続化（DiskSecondaryBTree）に伴い ~~2.8x~~ → **6.4x** に増加。インデックス検索自体が DiskSecondaryBTree のページフェッチ（B+Tree 走査 + PrefixScan）を必要とするため、インメモリ B-tree 時代より追加のオーバーヘッドが発生する。ただしこれはバッファプールにキャッシュ済みの場合の数値であり、実際のディスク I/O は発生していない。トレードオフとして、起動時の全行スキャンによるインデックス再構築が不要になる。

| 10,000行 | インデックスなし | 1,007,725 | 2,710,130 | **2.7x** |

**考察:** インデックスなしのフルスキャンは 2.7x と比較的安定。インデックスありは `GetByKeys` のバッチフェッチ最適化（ソート済みキーでリーフチェーン1パス走査）により **10.2x → 1.5x に大幅改善**。セカンダリインデックスのディスク永続化後も **1.4x** を維持。多数行ヒット (2,000行) ではプライマリ B+Tree のリーフ走査コストが支配的であり、セカンダリインデックスの検索コスト増は相対的に小さい。

#### 4b. ユニークカラム (WHERE val IN (50, 100, 150)) — 3行ヒット

| データ量 | 条件 | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|------|---------------|-------------|-------------|

### 7. ORDER BY + LIMIT

| パターン | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|---------------|-------------|-------------|
| PK ASC + LIMIT 10 | 1,504 | 4,592 | **3.1x** |
| PK DESC + LIMIT 10 | 1,464 | ~~18,916~~ 5,170 | ~~**13x**~~ **3.5x** |
| LIMIT 10 (ORDER BY なし) | ~~233,961~~ 1,200 | ~~1,871,012~~ 4,500 | ~~**8.0x**~~ **3.7x** |

**考察:** PK ASC + LIMIT は 3.1x で良好。PK DESC + LIMIT は `prevLeaf` バックポインタの追加により ~~13x~~ → **3.5x** にさらに改善。リーフページヘッダに `prevLeaf` (4B) を追加し、`findRightmostLeaf` (O(H)) + `prevLeaf` チェーン逆方向走査に変更したことで、LIMIT 10 なら H + 1〜2 ページフェッチで済むようになった。ASC の 3.1x とほぼ同等の水準に到達。改善の推移: 1,301x → 13x (リーフ PageID 収集) → **3.5x** (`prevLeaf` バックポインタ)。

LIMIT 10（ORDER BY なし）は `earlyLimit` を `ScanOrdered` に伝搬する最適化により ~~8.0x~~ → **3.7x** に改善。WHERE なし + LIMIT の場合、`scanSourceSingle` が `Scan()` (全行収集) の代わりに `ScanOrdered(name, false, limit)` を呼ぶことで、先頭リーフページから limit 行だけ読み取り早期打ち切りする。Memory 側も同じパスを通るため 233,961 → 1,200 ns/op に高速化。

### 8. JOIN (1,000行)

| パターン | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|---------------|-------------|-------------|
| WHERE なし + インデックスなし | 144,818 | 329,362 | **2.3x** |
| WHERE なし + インデックスあり | 96,074 | 385,764 | **4.0x** |
| 複合インデックス | 53,123 | 209,085 | **3.9x** |
| 個別インデックス | 79,747 | 378,921 | **4.8x** |
| LIMIT 10 | 11,896 | 36,034 | **3.0x** |

**考察:** JOIN ではインデックスなしの Hash Join パスが 2.3x で最も安定。インデックスありの場合はインナーテーブルの個別 Get が積算されるため 3.9〜4.8x に拡大する。LIMIT 10 では 3.0x と許容範囲。

## まとめ

| カテゴリ | Disk/Memory 比率 | 評価 |
|---------|-----------------|------|
| PK ポイントルックアップ | 3.5x | 良好 |
| セカンダリインデックス ポイントルックアップ | ~~2.8x~~ 6.4x | インデックス永続化のオーバーヘッド |
| フルスキャン (インデックスなし) | 2.7x〜4.1x | 許容範囲 |
| インデックス利用 + 少数ヒット (≤3行) | ~~2.8x〜4.4x~~ 6.3x〜9.2x | インデックス永続化のオーバーヘッド |
| インデックス利用 + 多数ヒット (100行〜) | ~~10.2x〜13.3x~~ ~~1.5x〜3.7x~~ 1.4x〜4.0x | 行フェッチが支配的で影響小 |
| IN インデックス (300行ヒット) | ~~12.9x~~ ~~2.4x~~ 3.1x | 行フェッチが支配的で影響小 |
| LIKE インデックス (疎キー) | ~~11.5x~~ ~~1.9x~~ 2.1x | 行フェッチが支配的で影響小 |
| ORDER BY PK ASC + LIMIT | 3.1x | 良好 |
| ORDER BY PK DESC + LIMIT | ~~13x~~ 3.5x | 改善済み (1,301x→13x→3.5x) |
| LIMIT (ORDER BY なし) | ~~8.0x~~ 3.7x | 改善済み (earlyLimit 伝搬) |
| JOIN (Hash Join) | 2.3x | 良好 |
| JOIN (インデックス利用) | 3.9x〜4.8x | 許容範囲 |

### 主要な知見

1. **PK ポイントルックアップの基本オーバーヘッドは 3.5x**: バッファプールがキャッシュ済みの場合、ページフェッチ + デコードのコストは一定。メモリストレージの Go ポインタ直接参照に比べて約 3.5 倍だが、データがメモリに収まらないケースを扱える利点がある。

2. **セカンダリインデックスのディスク永続化によるトレードオフ**: セカンダリインデックスを DiskSecondaryBTree（ディスク永続化 B+Tree）に変更したことで、ポイントルックアップは 2.8x → **6.4x** に増加した。これはインデックス検索自体がバッファプール経由のページフェッチ（B+Tree 走査 + PrefixScan）を必要とするためである。一方、起動時の全行スキャンによるインデックス再構築が不要になり、大きなテーブルでの起動時間が大幅に改善される。ヒット行数が多いクエリ（範囲検索、IN 300行、LIKE、複合インデックス）では行フェッチコストが支配的であり、インデックス永続化の影響は 0.3〜0.7x の微増に留まる。

3. **セカンダリインデックスからの多数行フェッチは改善済み**: `GetByKeys` のバッチフェッチ最適化により、rowKey をソートしてリーフチェーンを 1 回のシーケンシャルスキャンで処理する方式に変更。疎キー最適化（ヘッダースキップ + ギャップジャンプ）と部分デコード最適化（ページ内インライン走査 + マッチ時のみ DecodeRow）を組み合わせ、範囲検索は 10.2x → **1.4x**、IN（300行）は 12.9x → **3.1x**、LIKE は 11.5x → **2.1x**、複合インデックスは 13.3x → **4.0x** に改善。インデックス永続化後もこれらの数値はほぼ維持されている。

4. **`ForEachReverse` は `prevLeaf` バックポインタで ASC 同等に改善済み**: リーフページヘッダに `prevLeaf` (4B) バックポインタを追加し、`findRightmostLeaf` (O(H)) + `prevLeaf` チェーン逆方向走査に変更。ORDER BY PK DESC + LIMIT が 1,301x → 13x → **3.5x** に改善し、ASC (3.1x) とほぼ同等の水準に到達。LIMIT 10 なら木の高さ分 + 1〜2 ページフェッチで済む。

5. **LIMIT (ORDER BY なし) は `earlyLimit` 伝搬で改善済み**: `scanSourceSingle` が WHERE なし + LIMIT の場合に `ScanOrdered(name, false, limit)` を呼ぶことで、8.0x → **3.7x** に改善。DISTINCT + LIMIT は重複排除のため全行スキャンが必要なので 8.0x のまま。**改善案**: DISTINCT + LIMIT には `Scan()` のイテレータ化が必要。

6. **Hash Join は disk で最も効率的な JOIN パス**: inner テーブルのフルスキャン 1 回でハッシュテーブルを構築するため、個別 Get の積算が発生せず 2.3x に留まる。
