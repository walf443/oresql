# ディスクストレージ ベンチマークレポート

実行日: 2026-02-28
環境: Apple M2 Pro, darwin/arm64, Go

## 概要

disk ストレージ (`--storage disk`) と memory ストレージ (デフォルト) の性能差を、クエリパターン別に計測した。
disk ストレージはデータをページ単位でディスクに格納し、バッファプール (LRU 256 ページ) でホットページのみメモリに保持する。
memory ストレージのセカンダリインデックスはメモリ上の B-tree、disk ストレージのセカンダリインデックスは同一 `.db` ファイル内にページベースの B+Tree として永続化される（DiskSecondaryBTree）。disk ではインデックスの検索自体もバッファプール経由のページフェッチとなるが、インラインページスキャン最適化（ページバッファ上で直接エントリを走査し、Go ヒープへのアロケーションを排除）と `DecodeRowN` プリアロケーション最適化（カラム数を事前に渡して Row スライスの再割り当てを排除）により、ポイントルックアップのオーバーヘッドは Memory 比 2〜3x に抑えられている。

## ベンチマーク結果

### 1. PK / セカンダリインデックス ポイントルックアップ (10,000行)

| パターン | Memory (ns/op) | Disk (ns/op) | Disk/Memory | Memory (B/op) | Disk (B/op) |
|---------|---------------|-------------|-------------|--------------|------------|
| PK lookup (`WHERE id = X`) | 879 | 1,837 | **2.1x** | 920 | 4,280 |
| セカンダリインデックス (`WHERE val = X`) | 1,054 | 2,602 | **2.5x** | 1,050 | 4,554 |

**考察:** PK ルックアップは disk で 2.1x。セカンダリインデックスのルックアップはインラインページスキャン最適化により **2.5x** に改善（以前は 6.4x）。DiskSecondaryBTree の `findLeaf` / `PrefixScan` でページバッファ上の直接走査を行い、`decodeSecLeafPage` / `decodeSecInternalPage` の全エントリ分 `make+copy` を排除したことで、B+Tree 走査のアロケーションオーバーヘッドが大幅に削減された（14,770 B/op → 4,554 B/op）。さらに `DecodeRowN` プリアロケーションにより allocs/op が 2 削減（PK: 34→32, Sec: 42→40）、B/op も 48 削減された。

### 3. 範囲検索 (WHERE val >= X AND val <= Y, ヒット率 20%)

| データ量 | 条件 | Memory (ns/op) | Disk (ns/op) | Disk/Memory |

| 10,000行 | インデックスなし | 1,004,225 | 2,097,575 | **2.1x** |

**考察:** インデックスなしのフルスキャンは 2.1x。インデックスありは `GetByKeysSorted` のバッチフェッチ最適化（ソート済みキーでリーフチェーン1パス走査）+ `DecodeRowN` プリアロケーションにより **1.2x** に改善。多数行ヒット (2,000行) ではプライマリ B+Tree のリーフ走査コストが支配的であり、セカンダリインデックスの検索コスト増は相対的に小さい。

### 7. ORDER BY + LIMIT

| パターン | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|---------------|-------------|-------------|
| PK ASC + LIMIT 10 | 1,480 | 4,072 | **2.8x** |
| PK DESC + LIMIT 10 | 1,517 | 4,628 | **3.1x** |
| LIMIT 10 (ORDER BY なし) | 1,194 | 3,745 | **3.1x** |

**考察:** PK ASC + LIMIT は 2.8x で良好。PK DESC + LIMIT は `prevLeaf` バックポインタにより **3.1x**。LIMIT 10（ORDER BY なし）は `earlyLimit` を `ScanOrdered` に伝搬する最適化により **3.1x**。`DecodeRowN` プリアロケーションにより LIMIT パスでも allocs/op が削減されている。

## まとめ

| カテゴリ | Disk/Memory 比率 | 評価 |
|---------|-----------------|------|
| PK ポイントルックアップ | 2.1x | 良好 |
| セカンダリインデックス ポイントルックアップ | 2.5x | 良好（インライン最適化で改善） |
| フルスキャン (インデックスなし) | 2.1x〜3.2x | 許容範囲 |
| インデックス利用 + 少数ヒット (≤3行) | 1.7x〜2.5x | 良好（インライン最適化で改善） |
| インデックス利用 + 多数ヒット (100行〜) | 1.2x〜3.3x | 良好 |
| IN インデックス (300行ヒット) | 2.2x | 良好 |
| IN ユニーク (3行ヒット) | 2.0x | 良好（インライン最適化で改善） |
| LIKE インデックス (疎キー) | 1.6x | 良好 |
| ORDER BY PK ASC + LIMIT | 2.8x | 良好 |
| ORDER BY PK DESC + LIMIT | 3.1x | 良好 |
| LIMIT (ORDER BY なし) | 3.1x | 良好 |
| JOIN (Hash Join) | 1.3x〜1.9x | 良好 |
| JOIN (インデックス利用) | 1.3x〜2.1x | 良好 |

### 主要な知見

1. **PK ポイントルックアップの基本オーバーヘッドは 2.1x**: バッファプールがキャッシュ済みの場合、ページフェッチ + デコードのコストは一定。メモリストレージの Go ポインタ直接参照に比べて約 2.1 倍だが、データがメモリに収まらないケースを扱える利点がある。

2. **セカンダリインデックスのインラインページスキャン最適化**: DiskSecondaryBTree の読み取り系メソッド（`findLeaf`, `PrefixScan`, `RangeScan`, `ForEach` 等）でページバッファ上の直接走査を行い、`decodeSecLeafPage` / `decodeSecInternalPage` の全エントリ分 `make+copy` を排除。これによりポイントルックアップは 6.4x → **2.5x** に改善。B/op も 14,770 → 4,554 に削減された。

3. **`DecodeRowN` プリアロケーション最適化**: `DecodeRow` にカラム数を事前に渡す `DecodeRowN` を導入し、DiskBTree のホットパス（`Get`, `GetByKeysSorted`, `ForEach`, `ForEachReverse`, `ForEachRange`）で Row スライスの `append` による再割り当てを排除。全ベンチマークで一貫して allocs/op が 2 削減、B/op が 48 削減された。

4. **セカンダリインデックスからの多数行フェッチは改善済み**: `GetByKeys` のバッチフェッチ最適化により、rowKey をソートしてリーフチェーンを 1 回のシーケンシャルスキャンで処理する方式に変更。疎キー最適化（ヘッダースキップ + ギャップジャンプ）と部分デコード最適化（ページ内インライン走査 + マッチ時のみ DecodeRowN）を組み合わせ、範囲検索は **1.2x**、IN（300行）は **2.2x**、LIKE は **1.6x**、複合インデックスは **3.3x**。

5. **`ForEachReverse` は `prevLeaf` バックポインタで ASC 同等に改善済み**: リーフページヘッダに `prevLeaf` (4B) バックポインタを追加し、`findRightmostLeaf` (O(H)) + `prevLeaf` チェーン逆方向走査に変更。ORDER BY PK DESC + LIMIT が **3.1x** で ASC (2.8x) とほぼ同等の水準。

6. **LIMIT (ORDER BY なし) は `earlyLimit` 伝搬で改善済み**: `scanSourceSingle` が WHERE なし + LIMIT の場合に `ScanOrdered(name, false, limit)` を呼ぶことで **3.1x**。DISTINCT + LIMIT は重複排除のため全行スキャンが必要なので 5.7x。**改善案**: DISTINCT + LIMIT には `Scan()` のイテレータ化が必要。

7. **Hash Join は disk で最も効率的な JOIN パス**: inner テーブルのフルスキャン 1 回でハッシュテーブルを構築するため、個別 Get の積算が発生せず 1.3〜1.9x に留まる。
