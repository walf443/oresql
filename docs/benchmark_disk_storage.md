# ディスクストレージ ベンチマークレポート

実行日: 2026-02-27
環境: Apple M2 Pro, darwin/arm64, Go

## 概要

disk ストレージ (`--storage disk`) と memory ストレージ (デフォルト) の性能差を、クエリパターン別に計測した。
disk ストレージはデータをページ単位でディスクに格納し、バッファプール (LRU 256 ページ) でホットページのみメモリに保持する。
セカンダリインデックスは disk/memory 共にメモリ上の B-tree。

## ベンチマーク結果

### 1. PK / セカンダリインデックス ポイントルックアップ (10,000行)

| パターン | Memory (ns/op) | Disk (ns/op) | Disk/Memory | Memory (B/op) | Disk (B/op) |
|---------|---------------|-------------|-------------|--------------|------------|
| PK lookup (`WHERE id = X`) | 882 | 3,054 | **3.5x** | 920 | 8,008 |
| セカンダリインデックス (`WHERE val = X`) | 1,080 | 3,048 | **2.8x** | 1,050 | 8,138 |

**考察:** ポイントルックアップは disk で 3〜3.5x 遅くなる。主な原因はバッファプールのページフェッチ + 行デコードのオーバーヘッド。ページがバッファプールにキャッシュ済みのため、実際のディスク I/O は発生していない。メモリ割り当ては約 8x 増加（ページバッファのコピー分）。

---

### 2. 等値検索 (WHERE val = X)

| データ量 | 条件 | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|------|---------------|-------------|-------------|
| 1,000行 | インデックスなし | 46,310 | 189,575 | **4.1x** |
| 1,000行 | インデックスあり | 1,101 | 3,030 | **2.8x** |
| 10,000行 | インデックスなし | 605,591 | 2,294,906 | **3.8x** |
| 10,000行 | インデックスあり | 1,089 | 3,129 | **2.9x** |

**考察:** インデックスありの場合は 2.8〜2.9x の一定の差。インデックスなしのフルスキャンは全行のページデコードが必要なため 3.8〜4.1x に拡大する。

---

### 3. 範囲検索 (WHERE val >= X AND val <= Y, ヒット率 20%)

| データ量 | 条件 | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|------|---------------|-------------|-------------|
| 10,000行 | インデックスなし | 1,007,725 | 2,710,130 | **2.7x** |
| 10,000行 | インデックスあり | 419,190 | 4,293,241 | **10.2x** |

**考察:** インデックスなしのフルスキャンは 2.7x と比較的安定。一方、**インデックスありは 10.2x と大幅に悪化**する。これはセカンダリインデックスから取得した 2,000 件の rowKey に対して `DiskBTree.Get()` を個別に呼び出すためで、1 回のGet あたりページフェッチ + デコードのオーバーヘッドが積算される。ヒット件数が多い範囲検索ではフルスキャンの方が disk で有利になるケースがある。

### 4. IN 検索

#### 4a. 低カーディナリティ (WHERE category IN (5, 10, 15)) — 300行ヒット

| データ量 | 条件 | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|------|---------------|-------------|-------------|
| 10,000行 | インデックスなし | 818,128 | 2,482,200 | **3.0x** |
| 10,000行 | インデックスあり | 52,262 | 676,555 | **12.9x** |

#### 4b. ユニークカラム (WHERE val IN (50, 100, 150)) — 3行ヒット

| データ量 | 条件 | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|------|---------------|-------------|-------------|
| 10,000行 | インデックスなし | 642,609 | 2,383,775 | **3.7x** |
| 10,000行 | インデックスあり | 1,827 | 8,054 | **4.4x** |

**考察:** ヒット件数が少ない場合（3行）は 4.4x の一定差。ヒット件数が多い場合（300行）は 12.9x に拡大。範囲検索と同様、セカンダリインデックスからの個別 Get の積算コストが支配的。

### 5. LIKE 前方一致 (WHERE name LIKE 'name\_50%') — 11行ヒット

| データ量 | 条件 | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|------|---------------|-------------|-------------|
| 10,000行 | インデックスなし | 855,254 | 2,564,365 | **3.0x** |
| 10,000行 | インデックスあり | 20,706 | 238,238 | **11.5x** |

**考察:** 11行ヒットでも 11.5x の差。LIKE インデックススキャンは B-tree の範囲走査 + 個別 Get で構成されるため、ヒット件数以上のページアクセスが発生する。

### 6. 複合インデックス (WHERE category = X AND val >= Y)

| データ量 | 条件 | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|------|---------------|-------------|-------------|
| 10,000行 | インデックスなし | 717,594 | 2,365,550 | **3.3x** |
| 10,000行 | インデックスあり | 16,200 | 215,308 | **13.3x** |

**考察:** 複合インデックスでも個別 Get のオーバーヘッドにより 13.3x の差。

---

### 7. ORDER BY + LIMIT

| パターン | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|---------------|-------------|-------------|
| PK ASC + LIMIT 10 | 1,504 | 4,592 | **3.1x** |
| PK DESC + LIMIT 10 | 1,538 | 2,000,437 | **1,301x** |
| LIMIT 10 (ORDER BY なし) | 233,961 | 1,871,012 | **8.0x** |

**考察:** PK ASC + LIMIT は 3.1x で良好。**しかし PK DESC + LIMIT は 1,301x と壊滅的に遅い。** 現在の `ForEachReverse` 実装は全エントリをメモリに収集してから逆順にするため、LIMIT があっても全行スキャンが走る。`ForEach`（ASC）は B+Tree のリーフチェーンを辿るため効率的だが、`ForEachReverse` にはこの最適化がない。

LIMIT 10（ORDER BY なし）も 8.0x と大きめ。これは `Scan()` が全行をスライスに収集してから返すため。

---

### 8. JOIN (1,000行)

| パターン | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|---------------|-------------|-------------|
| WHERE なし + インデックスなし | 144,818 | 329,362 | **2.3x** |
| WHERE なし + インデックスあり | 96,074 | 385,764 | **4.0x** |
| 複合インデックス | 53,123 | 209,085 | **3.9x** |
| 個別インデックス | 79,747 | 378,921 | **4.8x** |
| LIMIT 10 | 11,896 | 36,034 | **3.0x** |

**考察:** JOIN ではインデックスなしの Hash Join パスが 2.3x で最も安定。インデックスありの場合はインナーテーブルの個別 Get が積算されるため 3.9〜4.8x に拡大する。LIMIT 10 では 3.0x と許容範囲。

---

### 9. DISTINCT + LIMIT

| パターン | Memory (ns/op) | Disk (ns/op) | Disk/Memory |
|---------|---------------|-------------|-------------|
| DISTINCT LIMIT 10 | 237,619 | 1,908,082 | **8.0x** |

**考察:** LIMIT 10 + ORDER BY なし でも 8.0x。Scan() の全行収集がボトルネック。

## まとめ

| カテゴリ | Disk/Memory 比率 | 評価 |
|---------|-----------------|------|
| ポイントルックアップ (PK / セカンダリ) | 2.8x〜3.5x | 良好 |
| フルスキャン (インデックスなし) | 2.7x〜4.1x | 許容範囲 |
| インデックス利用 + 少数ヒット (≤3行) | 2.8x〜4.4x | 良好 |
| インデックス利用 + 多数ヒット (100行〜) | 10.2x〜13.3x | 要改善 |
| ORDER BY PK ASC + LIMIT | 3.1x | 良好 |
| ORDER BY PK DESC + LIMIT | 1,301x | **要修正** |
| LIMIT (全行 Scan 経由) | 8.0x | 要改善 |
| JOIN (Hash Join) | 2.3x | 良好 |
| JOIN (インデックス利用) | 3.9x〜4.8x | 許容範囲 |

### 主要な知見

1. **ポイントルックアップの基本オーバーヘッドは 3x 程度**: バッファプールがキャッシュ済みの場合、ページフェッチ + デコードのコストは一定。メモリストレージの Go ポインタ直接参照に比べて約 3 倍だが、データがメモリに収まらないケースを扱える利点がある。

2. **セカンダリインデックスからの多数行フェッチが最大のボトルネック**: セカンダリインデックスで rowKey を取得した後、`DiskBTree.Get()` で 1 行ずつフェッチする際のオーバーヘッドが積算される。100行以上ヒットする場合は 10x〜13x の性能低下。**改善案**: `GetByKeys` で一括フェッチする際、rowKey をソートしてリーフチェーンを 1 回のシーケンシャルスキャンで処理する最適化が有効。

3. **`ForEachReverse` は全行収集のため壊滅的に遅い**: ORDER BY PK DESC + LIMIT で 1,301x の性能低下。**改善案**: リーフチェーンの全リーフ PageID を収集し、逆順にリーフを読み進める実装に変更する。メモリ消費は PageID のみ（4B × リーフ数）で済む。

4. **`Scan()` が全行をスライスに収集するため、LIMIT の恩恵が薄い**: ORDER BY なし + LIMIT や DISTINCT + LIMIT でも 8.0x の差。**改善案**: `Scan()` にイテレータ/コールバックパターンを導入し、必要な行数だけ読み取る。

5. **Hash Join は disk で最も効率的な JOIN パス**: inner テーブルのフルスキャン 1 回でハッシュテーブルを構築するため、個別 Get の積算が発生せず 2.3x に留まる。

## 改善優先度

| 改善項目 | 影響 | 難易度 |
|---------|------|-------|
| `ForEachReverse` をリーフ逆順走査に変更 | ORDER BY DESC が 1,301x → 3x に改善見込み | 中 |
| `GetByKeys` のバッチフェッチ最適化 | インデックス利用クエリが 10x → 3〜4x に改善見込み | 中 |
| `Scan()` のイテレータ化 / 早期打ち切り対応 | LIMIT クエリが 8x → 3x に改善見込み | 高 |
